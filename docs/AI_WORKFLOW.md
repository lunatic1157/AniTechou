# AniTechou AI 协作开发流程说明

## 1. 文档目的
本文档用于记录 AniTechou 的人机协作开发方法：在需求拆分、代码调研、实现迭代、验证回归与发布整理等环节中，如何使用 AI 提升效率与质量。
该文档面向开发与项目复盘，不作为终端用户使用说明。

## 2. 基本原则
- 人负责：产品定位、关键取舍、验收标准、安全边界与最终发布决策
- AI 负责：信息检索、结构化梳理、实现草案、重构建议、文档整理与风险提醒
- 所有输出必须可验证：功能以可运行产物和可复现步骤为准，文本说明不视为完成
- 变更必须可追踪：在关键节点通过测试、发布产物与文档落盘形成闭环

## 3. 协作流程（端到端）

### 3.1 需求拆解与边界定义
- 将目标拆分为可交付的小任务，明确输入/输出与验收标准
- 识别影响范围：涉及的视图、服务、配置、数据库、资源与发布流程
- 对非功能需求（性能、可用性、发布体验）设定最低达标线

### 3.2 代码库调研与定位
- 快速定位入口与调用链，形成“模块-责任-数据流”映射
- 优先识别风险点：数据落盘路径、发布依赖、异常处理、资源引用、UI 线程阻塞
- 在修改前确认仓库现有风格与约束，避免引入不一致实现

### 3.3 实现迭代与重构
- 在现有结构上做增量改动，优先复用已存在的服务与工具类
- 对跨文件改动保持一致性：命名、异常处理、路径规则、UI 行为
- 针对高风险区域（编辑器、导入导出、发布流程）优先写最小可回归验证

### 3.4 验证与回归
- 每次核心改动后至少通过：构建、测试、发布产物启动
- 对 UI 交互类改动（富文本编辑、拖拽、缩放）必须进行手动 smoke test
- 对路径与落盘行为进行检查，避免把配置/数据库写入程序目录

### 3.5 发布整理
- 发布前统一版本号来源，确保应用内版本、安装器版本、Release tag 对齐
- 发布说明强调差异化卖点（AI + 笔记）、安装方式、数据目录与已知事项
- 产物生成后在干净目录做一次启动验证，再进行上传与发布

## 4. 在 AniTechou 中的典型落点
- 调研阶段优先定位：配置、数据库、作品服务、笔记编辑器、AI 服务、发布脚本/安装器
- 实现阶段常见协作点：
  - 笔记编辑器交互增强（插图、保存、性能）
  - 主题与 UI 一致性整理
  - 导入规则与去重逻辑优化
  - 安装器与发布产物闭环
- 验证阶段常见检查点：
  - 数据目录固定为 LocalAppData，避免污染安装目录
  - 发布产物可启动、依赖完整、版本信息一致

## 5. 责任边界（明确不可交给 AI 的部分）
- 任何密钥与敏感信息不得进入仓库与日志
- UI 体验最终由人工评估：交互一致性、手感、边界情况
- 发布时机与版本策略由人工决定
- 对外表述由人工定调：重点、节奏与用词

## 6. 局限与风险控制
- AI 生成实现需要二次审查：线程模型、异常处理、性能开销与可维护性
- 富文本与拖拽类问题必须以真实交互结果为准，不能只看代码“看起来正确”
- 所有影响发布的变更必须在发布产物上复测，避免“Debug 可用，Release 不可用”

## 7. 结论
AniTechou 的开发采用“人主导 + AI 协作”的工作方式：以更低成本完成更系统的调研、实现、验证与发布整理，同时保持关键决策与风险控制由人工承担。
